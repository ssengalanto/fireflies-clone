# Implementation Plan: Fireflies Clone

**Branch**: `001-fireflies-clone` | **Date**: 2026-05-17 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-fireflies-clone/spec.md`

## Summary

A single-user, local-first meeting capture app: record audio in-browser, supply a transcript, then trigger an AI provider (Anthropic Claude, server-side proxied) to produce a streamed summary and a structured action-item list. Persisted client-side via a thin Next.js API façade. Built TDD-first in the strict order schema → store → hook → component, with the project's existing skills (`fireflies-swr`, `fireflies-tdd`, `fireflies-forms`, `fireflies-claude-api`) acting as the binding conventions.

The technical approach hinges on three clean lines:

1. **Server-state vs. UI-state**: SWR owns every byte that came from a network response (meetings, summaries, action items). Zustand owns every byte the user is composing or filtering (drafts, filters, modal/sidebar). The two never share keys.
2. **AI key never leaves the server**: `app/api/claude/route.ts` is the only file in the repo that imports `@anthropic-ai/sdk`. Browser code talks to `/api/claude/*` exclusively.
3. **Forms are a chain, not a soup**: zod schema → `zodResolver` → `useForm` → `useSWRMutation`. Every form has one hook that owns the whole submit pipeline; the component just renders fields.

## Technical Context

**Language/Version**: TypeScript 5.4+, Node.js 20+ (Next.js 14 minimum)
**Primary Dependencies**:
  - `next@^14` (App Router), `react@^18`, `react-dom@^18`
  - `swr@^2` (server state — bound `mutate`, `useSWRInfinite`, `useSWRMutation`)
  - `zustand@^4` (client UI state, with `persist` + `partialize`)
  - `react-hook-form@^7`, `zod@^3`, `@hookform/resolvers@^3`
  - `tailwindcss@^3`, shadcn/ui primitives (copied into `components/ui/`, not a runtime package)
  - `@anthropic-ai/sdk@^0.30+` (server-only — never imported from a client component)
**Storage**:
  - Client-side: browser `localStorage` for Zustand-persisted UI state (filters, draft, auth session)
  - Server-side: in-memory `Map<string, Meeting>` inside the API route module, seeded from `data/meetings.seed.json` on cold start. No external database. The API routes are the SWR fetcher target so all reads/writes flow through them.
**Testing**:
  - `jest@^29` + `@testing-library/react@^16` + `@testing-library/user-event@^14` + `@testing-library/jest-dom@^6`
  - `ts-jest@^29` (or `next/jest`) for TS transform
  - `jsdom` environment with a `MediaRecorder` polyfill (jsdom doesn't ship one)
  - Test wrappers per `fireflies-tdd/references/setup.md`
**Target Platform**: Latest two versions of Chrome, Safari, Firefox, Edge on desktop. Mobile is best-effort, not v1 acceptance.
**Project Type**: Web application (Next.js, fullstack in one project — App Router covers both the UI and the API routes).
**Performance Goals**:
  - Dashboard interactive in under 1 s for up to 200 meetings (SC-005)
  - Streaming summary's first visible token within 3 s on broadband (SC-002)
  - Cached summary/action-items render with no visible loading state on re-open (SC-003)
**Constraints**:
  - `ANTHROPIC_API_KEY` is server-only — never inlined into any `NEXT_PUBLIC_*` var, never imported from a client component (constitution gate)
  - TDD discipline is non-negotiable: every new module gets a failing test first, ordered schema → store → hook → component (constitution gate)
  - SWR owns server state; Zustand owns UI state; the two never duplicate keys (constitution gate)
  - Action-item parsing must tolerate markdown fences and default to `[]` on parse failure (FR-010, FR-011)
**Scale/Scope**: ~200 meetings per user on the device; single signed-in user; no cross-device sync; ~50 components total; ~10 forms; 2 AI endpoints.

**Structure Decision**: Single Next.js App Router project. The repo is monolithic — no separate backend service. The Next.js API routes (`app/api/**/route.ts`) play the role of "backend" but live in the same codebase. This matches the spec's local-first, single-device assumption and lets us share TypeScript types between client hooks and API handlers via `lib/schemas/` without a separate package.

The split between `lib/fetchers/` (no React, no SWR — plain `async` functions) and `lib/hooks/` (the only files that import `swr`) is deliberate: it keeps fetchers independently unit-testable and prevents accidental SWR usage from a component (the test-pattern in `fireflies-tdd` mocks the fetcher, never the hook).
